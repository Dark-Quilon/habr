# Требования: Django REST API

## Введение

Данный документ описывает функциональные и нефункциональные требования к REST API на базе Django REST Framework (DRF). API обеспечивает взаимодействие между существующим Django-бэкендом и фронтендом на Next.js. Существующая MTV-архитектура остаётся нетронутой — API добавляется параллельно.

## Требования

### 1. Инфраструктура и настройка

#### 1.1 Установка зависимостей
**Как** разработчик,
**я хочу** добавить `djangorestframework`, `django-cors-headers` и `django-filter` в `requirements.txt`,
**чтобы** иметь все необходимые пакеты для работы API.

**Критерии приёмки:**
- [ ] `requirements.txt` содержит `djangorestframework>=3.15`, `django-cors-headers>=4.3`, `django-filter>=23.5`
- [ ] Все пакеты устанавливаются без конфликтов через `pip install -r requirements.txt`

#### 1.2 Настройка Django settings
**Как** разработчик,
**я хочу** настроить `mysite/settings.py` для работы DRF, CORS и токен-аутентификации,
**чтобы** API корректно обрабатывал запросы от Next.js.

**Критерии приёмки:**
- [ ] `INSTALLED_APPS` содержит `rest_framework`, `rest_framework.authtoken`, `corsheaders`, `api`
- [ ] `CorsMiddleware` добавлен первым в `MIDDLEWARE`
- [ ] `CORS_ALLOWED_ORIGINS` включает `http://localhost:3000` и `http://127.0.0.1:3000`
- [ ] `CORS_ALLOW_CREDENTIALS = True`
- [ ] `REST_FRAMEWORK` настроен с `TokenAuthentication`, `SessionAuthentication`, `StandardPagination` (page_size=10), `DjangoFilterBackend`, `SearchFilter`, `OrderingFilter`

#### 1.3 Создание приложения `api/`
**Как** разработчик,
**я хочу** создать отдельное Django-приложение `api/` со структурой `serializers.py`, `views.py`, `urls.py`, `permissions.py`, `filters.py`,
**чтобы** изолировать API-логику от существующего MTV-кода.

**Критерии приёмки:**
- [ ] Приложение `api/` создано и зарегистрировано в `INSTALLED_APPS`
- [ ] Все API-маршруты доступны по префиксу `/api/v1/`
- [ ] Существующие MTV-маршруты (`blog/urls.py`) продолжают работать без изменений

### 2. Аутентификация

#### 2.1 Регистрация пользователя
**Как** новый пользователь,
**я хочу** зарегистрироваться через `POST /api/v1/auth/register/`,
**чтобы** получить токен и начать использовать API.

**Критерии приёмки:**
- [ ] `POST /api/v1/auth/register/` принимает `{username, password1, password2}`
- [ ] При успехе возвращает `201` с `{token, user: {id, username}}`
- [ ] При несовпадении паролей возвращает `400` с описанием ошибки
- [ ] При дублировании username возвращает `400`
- [ ] Автоматически создаётся `Profile` для нового пользователя
- [ ] Автоматически создаётся `Token` для нового пользователя

#### 2.2 Вход в систему
**Как** зарегистрированный пользователь,
**я хочу** войти через `POST /api/v1/auth/login/`,
**чтобы** получить токен для последующих запросов.

**Критерии приёмки:**
- [ ] `POST /api/v1/auth/login/` принимает `{username, password}`
- [ ] При успехе возвращает `200` с `{token, user: {id, username}}`
- [ ] При неверных credentials возвращает `400`
- [ ] Токен сохраняется в базе данных (модель `Token`)

#### 2.3 Выход из системы
**Как** авторизованный пользователь,
**я хочу** выйти через `POST /api/v1/auth/logout/`,
**чтобы** инвалидировать свой токен.

**Критерии приёмки:**
- [ ] `POST /api/v1/auth/logout/` требует `Authorization: Token <key>`
- [ ] При успехе удаляет токен из БД и возвращает `204`
- [ ] Без токена возвращает `401`

#### 2.4 Текущий пользователь
**Как** авторизованный пользователь,
**я хочу** получить данные о себе через `GET /api/v1/auth/me/`,
**чтобы** фронтенд мог отображать информацию о текущем пользователе.

**Критерии приёмки:**
- [ ] `GET /api/v1/auth/me/` возвращает `{id, username}` текущего пользователя
- [ ] Без токена возвращает `401`

### 3. Статьи

#### 3.1 Список статей
**Как** посетитель,
**я хочу** получить список опубликованных статей через `GET /api/v1/articles/`,
**чтобы** отображать их на главной странице Next.js.

**Критерии приёмки:**
- [ ] Возвращает пагинированный список `{count, next, previous, results}`
- [ ] Каждый элемент содержит: `id, slug, title, author, tags, status, views, rating, created_at`
- [ ] Анонимные пользователи видят только опубликованные статьи
- [ ] Поддерживает параметры: `?page=`, `?search=`, `?ordering=`, `?tags__slug=`, `?author__username=`
- [ ] `?search=` ищет по `title`, `content`, `tags__name`
- [ ] `?ordering=` поддерживает `created_at`, `-created_at`, `views`, `-views`, `rating`, `-rating`

#### 3.2 Детальная статья
**Как** посетитель,
**я хочу** получить полную статью через `GET /api/v1/articles/{slug}/`,
**чтобы** отображать её содержимое с комментариями.

**Критерии приёмки:**
- [ ] Возвращает все поля списка + `content`, `comments`, `updated_at`
- [ ] `comments` — вложенный список `[{id, author, content, created_at}]`
- [ ] Несуществующий slug возвращает `404`

#### 3.3 Создание статьи
**Как** авторизованный пользователь,
**я хочу** создать статью через `POST /api/v1/articles/`,
**чтобы** публиковать контент через Next.js-интерфейс.

**Критерии приёмки:**
- [ ] Принимает `{title, content, status, tags: ["tag1", "tag2"]}`
- [ ] `author` устанавливается автоматически из `request.user`
- [ ] `slug` генерируется автоматически из `title`
- [ ] Теги создаются автоматически если не существуют
- [ ] При успехе возвращает `201` с полными данными статьи
- [ ] Без токена возвращает `401`

#### 3.4 Редактирование статьи
**Как** автор статьи,
**я хочу** редактировать статью через `PUT/PATCH /api/v1/articles/{slug}/`,
**чтобы** обновлять контент.

**Критерии приёмки:**
- [ ] `PUT` требует все обязательные поля, `PATCH` — только изменяемые
- [ ] Только автор может редактировать (другие получают `403`)
- [ ] Теги обновляются при передаче нового списка
- [ ] Возвращает обновлённые данные статьи

#### 3.5 Удаление статьи
**Как** автор статьи,
**я хочу** удалить статью через `DELETE /api/v1/articles/{slug}/`,
**чтобы** управлять своим контентом.

**Критерии приёмки:**
- [ ] Только автор может удалить (другие получают `403`)
- [ ] При успехе возвращает `204`
- [ ] Без токена возвращает `401`

#### 3.6 Голосование за статью
**Как** авторизованный пользователь,
**я хочу** голосовать за статью через `POST /api/v1/articles/{slug}/vote/`,
**чтобы** выражать своё отношение к контенту.

**Критерии приёмки:**
- [ ] Принимает `{value: 1}` (лайк) или `{value: -1}` (дизлайк)
- [ ] Повторный голос с тем же значением снимает голос (toggle)
- [ ] Смена значения обновляет голос
- [ ] Автор не может голосовать за свою статью (возвращает `403`)
- [ ] Возвращает `{rating: int, user_vote: int | null}`

#### 3.7 Лента подписок
**Как** авторизованный пользователь,
**я хочу** получить ленту статей от подписанных авторов через `GET /api/v1/articles/feed/`,
**чтобы** следить за интересными авторами.

**Критерии приёмки:**
- [ ] Возвращает только статьи авторов, на которых подписан `request.user`
- [ ] Результат пагинирован
- [ ] Без токена возвращает `401`

### 4. Комментарии

#### 4.1 Список комментариев
**Как** посетитель,
**я хочу** получить комментарии к статье через `GET /api/v1/articles/{slug}/comments/`,
**чтобы** читать обсуждения.

**Критерии приёмки:**
- [ ] Возвращает список `[{id, author, content, created_at}]`
- [ ] Доступен без аутентификации

#### 4.2 Добавление комментария
**Как** авторизованный пользователь,
**я хочу** добавить комментарий через `POST /api/v1/articles/{slug}/comments/`,
**чтобы** участвовать в обсуждении.

**Критерии приёмки:**
- [ ] Принимает `{content}`
- [ ] `author` устанавливается из `request.user`
- [ ] Создаёт `Notification` для автора статьи (если автор ≠ комментатор)
- [ ] Возвращает `201` с данными комментария
- [ ] Без токена возвращает `401`

#### 4.3 Удаление комментария
**Как** автор комментария,
**я хочу** удалить свой комментарий через `DELETE /api/v1/articles/{slug}/comments/{id}/`,
**чтобы** управлять своими сообщениями.

**Критерии приёмки:**
- [ ] Только автор комментария может удалить (другие получают `403`)
- [ ] Возвращает `204`

### 5. Теги

#### 5.1 Список тегов
**Как** посетитель,
**я хочу** получить список всех тегов через `GET /api/v1/tags/`,
**чтобы** фронтенд мог отображать облако тегов.

**Критерии приёмки:**
- [ ] Возвращает `[{id, name, slug}]`
- [ ] Доступен без аутентификации
- [ ] Только чтение (создание тегов через статьи)

---

### 6. Профили

#### 6.1 Просмотр профиля
**Как** посетитель,
**я хочу** получить профиль пользователя через `GET /api/v1/profiles/{username}/`,
**чтобы** видеть информацию об авторе.

**Критерии приёмки:**
- [ ] Возвращает `{user, avatar, bio, followers_count, is_following}`
- [ ] `is_following` — `true` если `request.user` подписан на этого автора, иначе `false`
- [ ] Для анонимных пользователей `is_following = false`
- [ ] Несуществующий username возвращает `404`

#### 6.2 Редактирование своего профиля
**Как** авторизованный пользователь,
**я хочу** обновить свой профиль через `PATCH /api/v1/profiles/me/`,
**чтобы** изменять аватар и биографию.

**Критерии приёмки:**
- [ ] Принимает `{avatar, bio}` (оба поля опциональны)
- [ ] Обновляет только профиль `request.user`
- [ ] Поддерживает загрузку файла аватара (`multipart/form-data`)
- [ ] Без токена возвращает `401`

#### 6.3 Подписка / отписка
**Как** авторизованный пользователь,
**я хочу** подписаться или отписаться через `POST /api/v1/profiles/{username}/follow/`,
**чтобы** управлять своей лентой.

**Критерии приёмки:**
- [ ] Первый вызов создаёт `Follow` объект (подписка)
- [ ] Повторный вызов удаляет `Follow` объект (отписка)
- [ ] Нельзя подписаться на себя (возвращает `400`)
- [ ] Возвращает `{is_following: bool, followers_count: int}`
- [ ] Без токена возвращает `401`

### 7. Уведомления

#### 7.1 Список уведомлений
**Как** авторизованный пользователь,
**я хочу** получить свои уведомления через `GET /api/v1/notifications/`,
**чтобы** видеть активность по своим статьям.

**Критерии приёмки:**
- [ ] Возвращает уведомления только для `request.user`
- [ ] Каждое уведомление содержит `{id, actor, article, is_read, created_at}`
- [ ] Отсортированы по убыванию `created_at`
- [ ] Без токена возвращает `401`

#### 7.2 Отметить уведомления как прочитанные
**Как** авторизованный пользователь,
**я хочу** отметить все уведомления как прочитанные через `POST /api/v1/notifications/mark-read/`,
**чтобы** сбросить счётчик непрочитанных.

**Критерии приёмки:**
- [ ] Обновляет `is_read=True` для всех непрочитанных уведомлений `request.user`
- [ ] Возвращает `{marked: int}` — количество обновлённых уведомлений
- [ ] Без токена возвращает `401`

---

### 8. Нефункциональные требования

#### 8.1 Пагинация
- [ ] Все list-эндпоинты возвращают `{count, next, previous, results}`
- [ ] `page_size = 10` по умолчанию
- [ ] Поддерживается параметр `?page=N`

#### 8.2 Формат ошибок
- [ ] Все ошибки возвращаются в формате `{detail: "..."}` или `{field: ["..."]}`
- [ ] HTTP-коды соответствуют стандарту: 400 (валидация), 401 (нет аутентификации), 403 (нет прав), 404 (не найдено)

#### 8.3 CORS
- [ ] Все ответы содержат `Access-Control-Allow-Origin: http://localhost:3000` для запросов с этого origin
- [ ] Preflight OPTIONS-запросы обрабатываются корректно

#### 8.4 Производительность
- [ ] List-запросы используют `select_related` и `prefetch_related` для минимизации N+1 запросов
- [ ] Кэш Redis переиспользуется для list-эндпоинта статей
