# Документ требований

## Введение

Платформа для технических блогов, вдохновлённая habr.com, построенная на Django, PostgreSQL (или SQLite для разработки), Redis для кэширования и развёртываемая через Docker Compose. Платформа поддерживает написание статей в Markdown, аутентификацию пользователей, социальные функции (комментарии, голосование, подписки, уведомления) и поиск контента (поиск, теги, сортировка, пагинация). Весь стек запускается одной командой `docker compose up --build`.

## Глоссарий

- **Платформа** — веб-приложение «Клон Хабра» в целом
- **Web_Service** — контейнер Django/Gunicorn, обрабатывающий HTTP-запросы
- **Redis_Service** — контейнер Redis, обеспечивающий кэширование
- **DB_Service** — контейнер базы данных (PostgreSQL) или файл SQLite
- **Пользователь** — аутентифицированный человек с аккаунтом на Платформе
- **Автор** — Пользователь, создавший хотя бы одну Статью
- **Статья** — контент, написанный в Markdown, с заголовком, тегами и статусом публикации
- **Черновик** — Статья со статусом «draft», видимая только её Автору
- **Опубликованная_Статья** — Статья со статусом «published», видимая всем посетителям
- **Комментарий** — текстовый ответ, оставленный Пользователем на Опубликованной_Статье
- **Голос** — действие +1 или -1 от Пользователя на Опубликованной_Статье
- **Тег** — короткая метка, прикреплённая к Статье для категоризации
- **Профиль** — расширенная информация о пользователе, включая аватар и биографию
- **Подписка** — направленная подписка одного Пользователя на Автора
- **Уведомление** — внутриплатформенное оповещение, отправляемое Автору при добавлении Комментария к его Статье
- **Лента** — список Опубликованных_Статей от Авторов, на которых подписан Пользователь
- **Slug** — URL-безопасный уникальный идентификатор, производный от заголовка или имени
- **Кэш** — хранилище на базе Redis для снижения нагрузки на БД при частых чтениях
- **Dockerfile** — инструкции по сборке образа контейнера Web_Service
- **Compose_File** — файл `docker-compose.yml`, определяющий все сервисы
- **Env_File** — файл `.env`, содержащий секреты и конфигурацию для конкретного окружения

## Требования

### Требование 1: Настройка Docker-инфраструктуры

**Пользовательская история:** Как разработчик, я хочу, чтобы весь стек приложения был описан в Docker Compose, чтобы я мог запустить все сервисы одной командой.

#### Критерии приёмки

1. Compose_File ДОЛЖЕН определять три сервиса: `web`, `redis` и `db` (или `web` и `redis` с томом SQLite для начальной настройки).
2. КОГДА выполняется `docker compose up --build`, Платформа ДОЛЖНА стать доступной по адресу `http://localhost:8000` в течение 60 секунд.
3. Web_Service ДОЛЖЕН зависеть от Redis_Service, обеспечивая запуск Redis_Service до Web_Service.
4. Compose_File ДОЛЖЕН монтировать том хоста для медиафайлов, чтобы загруженные файлы сохранялись при перезапуске контейнеров.
5. Compose_File ДОЛЖЕН монтировать том хоста для файла базы данных SQLite, чтобы данные сохранялись при перезапуске контейнеров.
6. Web_Service ДОЛЖЕН читать всю конфигурацию из переменных окружения, поставляемых через Env_File.
7. ЕСЛИ Redis_Service недоступен при запуске, ТО Web_Service ДОЛЖЕН запуститься и обслуживать запросы, используя базу данных как запасной вариант, без сбоя.

---

### Требование 2: Dockerfile и образ контейнера

**Пользовательская история:** Как разработчик, я хочу готовый к продакшену Dockerfile, чтобы образ приложения был минимальным, воспроизводимым и быстро пересобираемым.

#### Критерии приёмки

1. Dockerfile ДОЛЖЕН использовать `python:3.12-slim` в качестве базового образа.
2. Dockerfile ДОЛЖЕН копировать `requirements.txt` до копирования исходного кода приложения, чтобы слой зависимостей кэшировался независимо от изменений кода.
3. Dockerfile ДОЛЖЕН выполнять `python manage.py collectstatic --noinput` на этапе сборки образа.
4. Dockerfile ДОЛЖЕН устанавливать `ENV PYTHONUNBUFFERED=1`, чтобы логи Django сразу появлялись в `docker compose logs`.
5. Dockerfile ДОЛЖЕН открывать порт 8000.
6. Dockerfile ДОЛЖЕН запускать приложение через Gunicorn с минимум 2 воркерами, привязанными к `0.0.0.0:8000`.
7. Платформа ДОЛЖНА включать файл `.dockerignore`, исключающий `.venv`, `__pycache__`, `*.pyc`, `.git`, `db.sqlite3`, `media/` и `Plan/` из контекста сборки.

---

### Требование 3: Конфигурация окружения

**Пользовательская история:** Как разработчик, я хочу, чтобы все секреты и значения, специфичные для окружения, хранились вне исходного кода, чтобы репозиторий было безопасно публиковать.

#### Критерии приёмки

1. Платформа ДОЛЖНА предоставлять файл `.env.example` со списком всех необходимых переменных окружения без реальных секретных значений.
2. Web_Service ДОЛЖЕН читать `SECRET_KEY`, `DEBUG`, `ALLOWED_HOSTS` и `REDIS_URL` из переменных окружения при запуске.
3. ЕСЛИ переменная окружения отсутствует, ТО Web_Service ДОЛЖЕН использовать безопасное значение по умолчанию, позволяющее продолжить локальную разработку.
4. Платформа ДОЛЖНА включать `.env` в `.gitignore`, чтобы реальные секреты никогда не попадали в систему контроля версий.
5. Web_Service ДОЛЖЕН разбивать переменную окружения `ALLOWED_HOSTS` по запятым для получения списка разрешённых хостов.

---

### Требование 4: Кэширование Redis

**Пользовательская история:** Как разработчик, я хочу, чтобы Опубликованные_Статьи и страницы списка статей кэшировались в Redis, чтобы повторные чтения не обращались к базе данных.

#### Критерии приёмки

1. КОГДА запрашивается Опубликованная_Статья, Web_Service ДОЛЖЕН отдавать её из Кэша, если запись кэша для Slug этой Статьи существует.
2. КОГДА Опубликованная_Статья отсутствует в Кэше, Web_Service ДОЛЖЕН получить её из базы данных и сохранить в Кэше с TTL 300 секунд.
3. КОГДА запрашивается страница списка статей, Web_Service ДОЛЖЕН отдавать её из Кэша, если запись кэша для данной комбинации порядка сортировки, фильтра по тегу и номера страницы существует.
4. КОГДА страница списка статей отсутствует в Кэше, Web_Service ДОЛЖЕН получить её из базы данных и сохранить в Кэше с TTL 60 секунд.
5. КОГДА Статья редактируется или удаляется, Web_Service ДОЛЖЕН инвалидировать запись Кэша для Slug этой Статьи и все записи Кэша списка статей.
6. КОГДА на Статью подаётся Голос, Web_Service ДОЛЖЕН инвалидировать запись Кэша для Slug этой Статьи.
7. КОГДА к Статье добавляется или удаляется Комментарий, Web_Service ДОЛЖЕН инвалидировать запись Кэша для Slug этой Статьи.
8. Redis_Service ДОЛЖЕН быть настроен с максимальным лимитом памяти 256 МБ и политикой вытеснения `allkeys-lru`.
9. Web_Service ДОЛЖЕН устанавливать `DJANGO_REDIS_IGNORE_EXCEPTIONS = True`, чтобы сбой Redis не вызывал ошибки HTTP 500.

---

### Требование 5: Аутентификация пользователей

**Пользовательская история:** Как посетитель, я хочу зарегистрироваться, войти и выйти, чтобы участвовать в работе платформы.

#### Критерии приёмки

1. КОГДА посетитель отправляет валидную форму регистрации, Платформа ДОЛЖНА создать аккаунт Пользователя, выполнить вход и перенаправить на страницу списка статей.
2. КОГДА посетитель отправляет невалидную форму регистрации, Платформа ДОЛЖНА повторно отобразить страницу регистрации с сообщениями об ошибках по полям.
3. КОГДА Пользователь отправляет валидные учётные данные для входа, Платформа ДОЛЖНА аутентифицировать Пользователя и перенаправить на страницу списка статей.
4. КОГДА Пользователь отправляет невалидные учётные данные для входа, Платформа ДОЛЖНА повторно отобразить страницу входа с сообщением об ошибке.
5. КОГДА Пользователь выходит из системы, Платформа ДОЛЖНА завершить сессию и перенаправить на страницу списка статей.
6. ПОКА Пользователь не аутентифицирован, Платформа ДОЛЖНА перенаправлять запросы к защищённым страницам на страницу входа.

---

### Требование 6: Управление статьями (CRUD)

**Пользовательская история:** Как Пользователь, я хочу создавать, читать, редактировать и удалять Статьи, чтобы публиковать свой контент и управлять им.

#### Критерии приёмки

1. КОГДА аутентифицированный Пользователь отправляет валидную форму создания статьи, Платформа ДОЛЖНА сгенерировать уникальный Slug из заголовка, сохранить Статью как Черновик или Опубликованную_Статью согласно выбранному статусу и перенаправить на страницу детали Статьи.
2. КОГДА заголовок уже имеет соответствующий Slug в базе данных, Платформа ДОЛЖНА добавить числовой суффикс для получения уникального Slug.
3. КОГДА посетитель запрашивает Опубликованную_Статью, Платформа ДОЛЖНА отображать заголовок Статьи, имя пользователя автора, дату создания, количество просмотров, рейтинг, теги, контент в формате Markdown и комментарии.
4. КОГДА посетитель, не являющийся Автором Статьи, просматривает Опубликованную_Статью, Платформа ДОЛЖНА увеличить счётчик просмотров Статьи на 1.
5. КОГДА аутентифицированный Пользователь отправляет валидную форму редактирования своей Статьи, Платформа ДОЛЖНА обновить Статью и перенаправить на страницу детали Статьи.
6. КОГДА аутентифицированный Пользователь запрашивает удаление своей Статьи и подтверждает его, Платформа ДОЛЖНА удалить Статью и перенаправить на страницу списка статей.
7. ЕСЛИ Пользователь пытается редактировать или удалить Статью, которой он не владеет, ТО Платформа ДОЛЖНА вернуть ответ HTTP 404.
8. ПОКА Пользователь не аутентифицирован, Платформа НЕ ДОЛЖНА отображать элементы управления созданием статей ни на одной странице.

---

### Требование 7: Теги

**Пользовательская история:** Как Пользователь, я хочу прикреплять теги к Статьям и фильтровать список статей по тегу, чтобы контент был организован по темам.

#### Критерии приёмки

1. КОГДА Пользователь вводит список тегов через запятую в форме статьи, Платформа ДОЛЖНА создать или получить каждый Тег и связать его со Статьёй.
2. КОГДА посетитель нажимает на ссылку Тега в Статье, Платформа ДОЛЖНА отобразить отфильтрованный список статей, содержащий только Опубликованные_Статьи, связанные с этим Тегом.
3. Платформа ДОЛЖНА отображать ссылки на Теги на каждой карточке Статьи в списке статей и на странице детали Статьи.

---

### Требование 8: Комментарии

**Пользовательская история:** Как Пользователь, я хочу оставлять и удалять комментарии к Опубликованным_Статьям, чтобы участвовать в обсуждении.

#### Критерии приёмки

1. КОГДА аутентифицированный Пользователь отправляет непустую форму комментария к Опубликованной_Статье, Платформа ДОЛЖНА сохранить Комментарий и перенаправить обратно на страницу детали Статьи.
2. КОГДА Пользователь просматривает Опубликованную_Статью, Платформа ДОЛЖНА отображать все Комментарии в хронологическом порядке с именем пользователя автора и временной меткой.
3. КОГДА аутентифицированный Пользователь запрашивает удаление своего Комментария и подтверждает его, Платформа ДОЛЖНА удалить Комментарий и перенаправить обратно на страницу детали Статьи.
4. ЕСЛИ Пользователь пытается удалить Комментарий, которым он не владеет, ТО Платформа ДОЛЖНА вернуть ответ HTTP 404.
5. ПОКА Пользователь не аутентифицирован, Платформа НЕ ДОЛЖНА отображать форму отправки комментария.

---

### Требование 9: Голосование

**Пользовательская история:** Как Пользователь, я хочу голосовать «за» или «против» Опубликованных_Статей, чтобы качественный контент выделялся через рейтинги.

#### Критерии приёмки

1. КОГДА аутентифицированный Пользователь, не являющийся Автором Статьи, голосует за Опубликованную_Статью, Платформа ДОЛЖНА записать Голос и перенаправить обратно на страницу детали Статьи.
2. КОГДА Пользователь подаёт тот же Голос, что и ранее поданный им на Статью, Платформа ДОЛЖНА удалить существующий Голос (переключение).
3. КОГДА Пользователь подаёт Голос, отличный от его существующего Голоса на Статью, Платформа ДОЛЖНА обновить Голос до нового значения.
4. Платформа ДОЛЖНА отображать рейтинг Статьи как количество голосов «за» минус количество голосов «против».
5. ЕСЛИ Автор Статьи пытается проголосовать за свою Статью, ТО Платформа ДОЛЖНА перенаправить на страницу детали Статьи без записи Голоса.
6. ПОКА Пользователь не аутентифицирован, Платформа НЕ ДОЛЖНА отображать элементы управления голосованием.

---

### Требование 10: Профили пользователей

**Пользовательская история:** Как Пользователь, я хочу иметь страницу профиля с аватаром и биографией, чтобы другие пользователи могли узнать обо мне.

#### Критерии приёмки

1. КОГДА посетитель запрашивает страницу профиля по имени пользователя, Платформа ДОЛЖНА отображать аватар Пользователя, биографию и список Опубликованных_Статей, написанных этим Пользователем.
2. КОГДА аутентифицированный Пользователь отправляет валидную форму редактирования профиля, Платформа ДОЛЖНА сохранить обновлённые аватар и биографию и перенаправить на страницу профиля Пользователя.
3. Платформа ДОЛЖНА автоматически создавать запись Профиля при регистрации нового Пользователя.
4. КОГДА загружается изображение аватара, Платформа ДОЛЖНА хранить его в поддиректории `avatars/` корня медиафайлов.

---

### Требование 11: Подписки и лента

**Пользовательская история:** Как Пользователь, я хочу подписываться на Авторов и видеть персонализированную ленту их статей, чтобы следить за интересующим меня контентом.

#### Критерии приёмки

1. КОГДА аутентифицированный Пользователь нажимает кнопку подписки на профиле другого Пользователя, Платформа ДОЛЖНА создать запись Подписки и перенаправить обратно на страницу этого профиля.
2. КОГДА аутентифицированный Пользователь, уже подписанный на Автора, снова нажимает кнопку подписки, Платформа ДОЛЖНА удалить запись Подписки (переключение) и перенаправить обратно на страницу профиля.
3. ЕСЛИ Пользователь пытается подписаться на себя, ТО Платформа ДОЛЖНА перенаправить на страницу профиля без создания записи Подписки.
4. Платформа ДОЛЖНА отображать количество подписчиков на каждой странице профиля.
5. КОГДА аутентифицированный Пользователь запрашивает страницу ленты, Платформа ДОЛЖНА отображать Опубликованные_Статьи от Авторов, на которых подписан Пользователь, упорядоченные по дате создания по убыванию, с пагинацией по 10 статей на страницу.

---

### Требование 12: Уведомления

**Пользовательская история:** Как Автор, я хочу получать уведомления, когда кто-то комментирует мою Статью, чтобы я мог своевременно ответить.

#### Критерии приёмки

1. КОГДА Пользователь оставляет Комментарий к Опубликованной_Статье, Платформа ДОЛЖНА создать Уведомление для Автора Статьи, если комментатор не является Автором.
2. КОГДА аутентифицированный Пользователь просматривает страницу уведомлений, Платформа ДОЛЖНА отображать все Уведомления для этого Пользователя, упорядоченные по дате создания по убыванию, и отмечать все непрочитанные Уведомления как прочитанные.
3. Платформа ДОЛЖНА отображать количество непрочитанных Уведомлений в навигационной панели для аутентифицированных Пользователей.
4. КОГДА количество непрочитанных Уведомлений равно нулю, Платформа НЕ ДОЛЖНА отображать числовой значок в навигационной панели.

---

### Требование 13: Поиск статей (поиск, сортировка, пагинация)

**Пользовательская история:** Как посетитель, я хочу искать, сортировать и листать список статей, чтобы эффективно находить нужный контент.

#### Критерии приёмки

1. КОГДА посетитель отправляет поисковый запрос, Платформа ДОЛЖНА возвращать Опубликованные_Статьи, заголовок, контент или теги которых содержат строку запроса (без учёта регистра), без дублирования.
2. КОГДА посетитель выбирает порядок сортировки «новые», Платформа ДОЛЖНА упорядочивать Опубликованные_Статьи по дате создания по убыванию.
3. КОГДА посетитель выбирает порядок сортировки «рейтинг», Платформа ДОЛЖНА упорядочивать Опубликованные_Статьи по вычисленному рейтингу (голоса «за» минус голоса «против») по убыванию, с датой создания в качестве дополнительного критерия.
4. КОГДА посетитель выбирает порядок сортировки «популярные», Платформа ДОЛЖНА упорядочивать Опубликованные_Статьи по количеству просмотров по убыванию, с датой создания в качестве дополнительного критерия.
5. Платформа ДОЛЖНА отображать 10 Опубликованных_Статей на страницу в списке статей.
6. КОГДА посетитель переходит на следующую страницу, Платформа ДОЛЖНА сохранять активный поисковый запрос, фильтр по тегу и порядок сортировки в ссылках пагинации.

### Требование 15: Статические файлы

**Пользовательская история:** Как разработчик, я хочу, чтобы статические файлы собирались и корректно обслуживались внутри Docker, чтобы CSS и JavaScript загружались в продакшене.

#### Критерии приёмки

1. Dockerfile ДОЛЖЕН выполнять `collectstatic` при сборке образа, помещая файлы в `STATIC_ROOT`.
2. Web_Service ДОЛЖЕН определять `STATIC_ROOT` как `BASE_DIR / 'staticfiles'`, чтобы у `collectstatic` была валидная целевая директория.
3. Compose_File ДОЛЖЕН монтировать или обслуживать директорию `staticfiles`, чтобы Gunicorn мог отдавать статические ресурсы.

### Требование 16: Генерация Slug и целостность обратного преобразования

**Пользовательская история:** Как разработчик, я хочу, чтобы генерация Slug была детерминированной и обратимой, чтобы URL статей были стабильными и уникальными.

#### Критерии приёмки

1. КОГДА заголовок передаётся генератору Slug, Платформа ДОЛЖНА создать Slug, содержащий только строчные ASCII-буквы, цифры и дефисы.
2. КОГДА один и тот же заголовок передаётся дважды, Платформа ДОЛЖНА создать два различных Slug, добавив числовой суффикс ко второму.
3. ДЛЯ ВСЕХ валидных заголовков Статей генерация Slug с последующим поиском Статьи по этому Slug ДОЛЖНА возвращать исходную Статью (свойство обратного преобразования).
