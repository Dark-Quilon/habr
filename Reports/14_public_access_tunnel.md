# Отчёт: Публичный доступ к сайту через туннель

## Обзор

Настройка публичного доступа к сайту с локального ноутбука (ноут = сервер) через туннель, чтобы любой человек мог открыть сайт по ссылке с телефона или компьютера.

---

## Проблемы и решения

### 1. ngrok заблокирован

**Проблема**: ngrok недоступен без VPN — IP заблокирован (санкции). Даже с токеном выдаёт `ERR_NGROK_9040: We do not allow agents to connect from your IP address`.

**Попытка**: winget install ngrok → ошибка 500 при скачивании. Скачан вручную `ngrok.exe`.

**Итог**: ngrok не подходит без VPN.

---

### 2. Cloudflare Tunnel нестабилен

**Проблема**: cloudflared постоянно падал с ошибкой:
```
ERR failed to dial a quic connection: timeout: no recent network activity
```

**Причина**: QUIC (UDP) протокол блокируется роутером или провайдером.

**Попытка**: запуск с `--protocol http2` — туннель регистрировался, но сразу отваливался с ошибкой `530 The origin has been unregistered from Argo Tunnel`.

**Итог**: cloudflared не подходит на данной сети.

---

### 3. localtunnel — рабочее решение

**Инструмент**: `npx localtunnel --port 3000`

**Установка**: не требуется, запускается через npx (Node.js уже установлен).

**Особенность**: при первом открытии показывает страницу подтверждения — нужно ввести публичный IP (указан прямо на странице).

**Итог**: работает стабильно, URL вида `https://xxx.loca.lt`.

---

## Архитектурная проблема: фронт и бэк на разных портах

**Проблема**: фронт (Next.js) на порту 3000, бэкенд (Django) на порту 8000. В `frontend/.env.local` был прописан абсолютный адрес:
```
NEXT_PUBLIC_API_URL=http://127.0.0.1:8000/api/v1
```

Когда друг открывал туннель на 3000 — его браузер пытался достучаться до `127.0.0.1:8000` на своём компьютере. Результат: "Not found".

**Решение**: настроен Next.js rewrite proxy в `next.config.js`:

```js
async rewrites() {
  return [
    {
      source: '/api/v1/:path*',
      destination: 'http://127.0.0.1:8000/api/v1/:path*',
    },
  ]
}
```

И разделены URL для серверного и клиентского кода в `frontend/src/lib/api.ts`:

```ts
// Для серверных компонентов Next.js (на ноуте) — абсолютный URL
export const SERVER_API_BASE = process.env.API_URL ?? 'http://127.0.0.1:8000/api/v1'

// Для браузера клиента — относительный URL (проксируется через Next.js)
export const API_BASE = process.env.NEXT_PUBLIC_API_URL ?? '/api/v1'
```

`frontend/.env.local`:
```
NEXT_PUBLIC_API_URL=/api/v1
API_URL=http://127.0.0.1:8000/api/v1
```

**Итог**: один туннель на порт 3000 — и фронт и бэкенд работают для внешнего пользователя.

## Итоговая схема работы

```
Телефон друга
     │
     ▼
localtunnel серверы (Европа)
     │
     ▼
Ноутбук: Next.js :3000
     │  (rewrite /api/v1/*)
     ▼
Ноутбук: Django :8000
     │
     ▼
SQLite + Redis
```

## Изменённые файлы

| Файл | Что изменено |
|------|-------------|
| `frontend/.env.local` | `NEXT_PUBLIC_API_URL=/api/v1`, добавлен `API_URL` |
| `frontend/next.config.js` | Добавлен `rewrites()` для проксирования API |
| `frontend/src/lib/api.ts` | Добавлен `SERVER_API_BASE`, `serverFetch` использует его |
| `.env` | `ALLOWED_HOSTS=*` для приёма запросов с любого домена |

## Как запустить

```powershell
# 1. Бэкенд
docker-compose up -d

# 2. Фронт (в отдельном терминале)
cd frontend
npm run build
npm run start

# 3. Туннель (в отдельном терминале)
npx localtunnel --port 3000
```

Открыть URL из вывода localtunnel, ввести IP с страницы подтверждения — сайт доступен.

## Почему медленно

Задержка складывается из:

| Источник | Задержка |
|----------|----------|
| Туннель (трафик через серверы в Европе) | ~100–300 мс |
| Upload скорость домашнего интернета | зависит от провайдера |
| Django обработка запроса | ~5–20 мс |
| Next.js SSR рендеринг | ~10–50 мс |

Основной виновник — туннель. Для нормальной скорости нужен VPS с прямым IP.

---

## Django vs C# по скорости

Замена бэкенда на ASP.NET Core **не повлияла бы на скорость** в данной конфигурации.

| | Django | ASP.NET Core |
|--|--------|-------------|
| Время ответа API | ~5–20 мс | ~1–5 мс |
| Разница | — | ~15 мс быстрее |

Разница в 15 мс теряется на фоне 200 мс задержки туннеля — её не заметно. C# дал бы ощутимый прирост только на реальном сервере с низкой латентностью сети, где каждая миллисекунда важна (высоконагруженные API, тысячи запросов в секунду).
